PART 5 · PRACTICAL GUIDE

OpenClaw Best Practices

From setup to production — the guide we wish existed on day one

OpenClaw is powerful out of the box, but getting it to work reliably takes intention. This guide covers everything from installation to multi-agent orchestration — the complete playbook for running a personal AI agent.

01

Install: Local vs. Cloud

First decision — where does your agent live?

Before configuring anything, you need to decide where OpenClaw runs. There are two approaches — and many people use both.

☁️ CLOUD (VPS / ALWAYS-ON)
Run on a Server
Deploy OpenClaw on a VPS (DigitalOcean, Hetzner, AWS). The agent runs 24/7 — always listening on Discord, proactively running heartbeats, monitoring your email. This is the "always-on assistant" setup. Requires basic server admin skills (SSH, Docker, systemd).
Best for: persistent agents, heartbeat monitoring, team use
💻 LOCAL (YOUR MACHINE)
Run on Your Computer
Run OpenClaw locally with docker compose up. The agent runs while your machine is on. Two big advantages: Claude doesn't allow OpenClaw to use OAuth — you need API tokens on the server, but locally you can use your Claude subscription plan directly. And crucially, you can use Claude Code (local CLI) to configure and fix OpenClaw — because a doctor can't treat itself.
Best for: setup, config, fixing broken agents, using subscription plan
💡 PRO TIP: USE LOCAL CLAUDE TO CONFIGURE OPENCLAW
Run Claude Code (Anthropic's local CLI agent) in your OpenClaw workspace folder. Ask it to write your SOUL.md, fix YAML configs, debug Docker issues, review memory files — all in natural language. Since Claude Code runs on your subscription plan (no API tokens needed), it costs nothing extra. And because it's a separate agent, it can fix problems that OpenClaw itself created — broken configs, corrupted memory, misconfigured skills. You use one AI to build and maintain the other.
🔄 THE COMMON SETUP
Cloud for running, local for fixing. Deploy your agent on a VPS so it's always on. But when OpenClaw messes up its own config — and it will — it can't fix itself. That's where local Claude comes in. Run Claude Code in your OpenClaw workspace folder to diagnose errors, rewrite broken configs, clean up corrupted memory files, and test new skills. A doctor can't treat itself — you need a separate AI to fix your AI.
PICKING YOUR INTERFACE
02

Choosing the Right Channel

Where do you talk to your agent? This matters more than you'd think.

OpenClaw connects to your agent through a messaging channel — the app where you send and receive messages. The most common options are Discord, Slack, Telegram, and Lark. Not all channels are created equal.

🏆 RECOMMENDED
Discord
Discord is bot-native — it was built from the ground up to support bots alongside human users. Creating a bot takes minutes, the API has generous rate limits, and you get features like threads, reactions, slash commands, and multiple channels out of the box. Most importantly for multi-agent setups: you can run multiple bots in the same server, each in their own channel, and they can even talk to each other.
Rate limits: 50 requests/second per bot — more than enough
ALTERNATIVES
Other Channels
Slack — Great if your team already lives in Slack. Good bot support, but the free tier limits message history and bot interactions.
Telegram — Simple setup, works well on mobile. Good for personal use. Limited threading and channel management compared to Discord.
Lark — Tempting if you use it for work, but Lark's API rate limits are too restrictive for an always-on agent. Frequent throttling makes it unreliable for anything beyond light use.
💡 WHY DISCORD WINS FOR MULTI-AGENT
If you're planning to run multiple agents (see Section 08), Discord is the clear winner. You can create a private server with channels like #research, #email, #coding, and #conductor — each with its own bot. Agents can read each other's channels, pass tasks between themselves, and you can watch the whole orchestra work in real time. No other platform makes this as easy.

Choosing the Right Model

OpenClaw doesn't have its own brain — you pick the LLM. Here's how to choose.

OpenClaw supports any LLM via API — Claude, GPT, DeepSeek, Gemini, or even local models. The best setup isn't one model for everything — it's different models for different tasks. OpenClaw lets you configure this per-task.

💡 RECOMMENDED SETUP: 4-MODEL ROUTING
The best practice is to assign the right model for each job: Claude Opus as the brain for research, analysis, and complex decisions. Claude Sonnet for daily chat — email, scheduling, writing. GPT-5.3 Codex for all coding tasks. Gemini Flash for fast execution — simple tool calls, quick follow-ups, and lightweight tasks. This 4-model setup gives you top-tier quality where it matters and keeps costs manageable by using cheaper models for routine work.
DEFINING YOUR AGENT
04

Writing a Great SOUL.md

Your agent is only as good as its soul — here's how to write one that works

SOUL.md is loaded at the start of every conversation. It defines your agent's personality, values, communication style, and boundaries. Most effective souls are 50–150 lines. Shorter and the agent lacks direction; longer and it starts ignoring things.

✅ DO THIS
  • Be specific: "Reply in 2–3 sentences max" beats "be concise"
  • Define boundaries: "Never send an email without asking me first"
  • Set tone: "Skip filler phrases like 'Great question!' — just answer"
  • Prioritize values: "You value: accuracy > speed > politeness"
  • Include context: "I work in fintech. Assume I know what an API is"
  • Iterate: Have it draft an email, give feedback, tweak the soul
❌ DON'T DO THIS
  • Vague instructions: "Be helpful and friendly" — every LLM already does this
  • Contradictions: "Be extremely thorough" + "Keep replies short"
  • Overloading: 500+ line souls get ignored past the first few sections
  • No personality: A generic soul = a generic assistant. Have opinions.
  • Forgetting boundaries: Without explicit limits, the agent will try everything
  • Set and forget: Your soul should evolve as you learn what works

EXAMPLE: A well-structured SOUL.md

# Soul

You are a sharp, direct assistant for a senior engineer.
Never use corporate jargon or filler phrases.
Prefer short answers (2-3 sentences) unless I ask for detail.

# Values
accuracy > speed > politeness
When unsure, say so. Don't guess.

# Boundaries
Never send emails, messages, or calendar invites without explicit approval.
Never delete files. Move to a "to-delete" folder instead.
If a task involves spending money, always ask first.

# Communication
Default language: English. Greet in Korean (안녕하세요).
Format: use markdown for structure, but keep it minimal.
If I send a single word like "email" or "calendar", treat it as
"check my email" or "what's on my calendar today".

# Context
I work at a tech company. I know programming concepts.
My manager is Lisa. My team works on the payments platform.
Our sprint ends every other Friday.
SETTING UP SKILLS
05

Skills: What to Install and How

ClawHub has 13,700+ skills — here's how to pick the right ones safely

A skill is a knowledge file — a set of best practices, instructions, and examples that teach your agent how to do something well. Unlike tools (which give the agent new abilities like "search the web"), skills give it expertise. Think of it this way: a tool is a hammer, a skill is knowing which nail to hit and how hard. (For a deeper explanation, see Part 0: Vocabulary and Part 3: Agents.)

OpenClaw ships with 53 bundled skills that are safe to use immediately. Beyond that, ClawHub is the community skill marketplace with 13,700+ skills. Think of it like an app store — powerful, but not everything on it is trustworthy.

Browse and search skills at clawhub.ai/skills (sorted by downloads)

🏷️ START WITH OFFICIAL SKILLS
Always install official skills first — the ones created by Peter Steinberger (OpenClaw's creator). These are battle-tested, regularly updated, and guaranteed not to contain malicious code. Look for the verified badge on ClawHub. Third-party community skills can be great too, but vet them before installing.

Our recommended starter kit

📧
ESSENTIAL · OFFICIAL
GOG (Google Workspace)
The most downloaded skill on ClawHub. Gives your agent access to Gmail, Calendar, Drive, Contacts, Sheets, and Docs. One skill covers most daily workflows. Install this first.
🔍
ESSENTIAL · OFFICIAL
Tavily Web Search
A search tool built specifically for AI agents. Fast, reliable web search that returns structured data the agent can work with. Essential for any research, fact-checking, or information-gathering task.
🔎
ESSENTIAL · OFFICIAL
Find Skills
Lets your agent search ClawHub and install new skills by itself. When asked to do something it doesn't know how, the agent can find and load the right skill on the fly. Meta but powerful.
🌐

Puppeteer-based browser control. Your agent can navigate websites, fill forms, compare prices, and extract data. Slower than API-based skills, but essential for tasks where no direct API exists.
🧠
RECOMMENDED · OFFICIAL
Self-Improving Agent
The agent analyzes its own mistakes and automatically updates its SOUL.md and memory to prevent repeating them. Like a built-in feedback loop. Especially useful during the first few weeks of setup.
🛠️
RECOMMENDED · OFFICIAL
Skill Creator
Write, test, and publish your own skills. When you find yourself repeating the same instructions, turn them into a reusable skill. This is how the ClawHub ecosystem was built — by users creating skills for their own workflows.
💡 THE 80/20 RULE
GOG + Tavily + Find Skills covers about 80% of what most people need an agent to do. Start with those three, use the agent for a week, and then add more skills as you hit specific gaps. Don't install 20 skills on day one — each skill adds to the prompt size and can slow things down.
✅ SKILL SAFETY
  • Start with official skills — created by Peter Steinberger, verified on ClawHub
  • Read the code before enabling any third-party skill
  • Check download count and reviews on ClawHub
  • Watch for typosquatting — "gogle-workspace" vs "google-workspace"
⚠️ SKILL RISKS
  • ClawHavoc attack: Hundreds of malicious skills were found on ClawHub using fake names
  • Hidden scripts in skills can exfiltrate SSH keys, API tokens, and cookies
  • No mandatory code review for published skills — trust is community-based
  • Treat third-party skills like npm packages — useful but verify first
UNDER THE HOOD
05b

What a Skill Looks Like

Skills aren't magic — they're just text files. Here's how they work and how to write your own.

A skill is a folder containing a SKILL.md file — a plain text document with instructions that get loaded into the agent's prompt when the skill is activated. That's it. No compiled code, no executables, no APIs. Just a markdown file that tells the LLM how to do something well.

ANATOMY OF A SKILL

# my-email-skill/
# SKILL.md ← the actual skill
# README.md ← description for ClawHub
# examples/ ← optional examples

---
name: email-drafting
description: Draft professional emails
triggers: [email, draft, compose, write]
version: 1.2.0
author: peter-steinberger
---

# Email Drafting Skill

When asked to draft an email:
1. Ask who it's for, the topic, and the tone
2. Write a subject line first (max 8 words)
3. Keep the body under 150 words
4. Never use "I hope this finds you well"
5. End with a clear call to action

# Tone Guidelines
- **Manager**: direct, concise, bullet points
- **Client**: warm, professional, solution-focused
- **Team**: casual, action-oriented

# Examples
[See examples/ folder for samples]
FRONTMATTER (YAML header)
The --- block at the top. Contains metadata: name, description, trigger words, version, author. OpenClaw uses triggers to decide when to activate this skill — if a user says "draft an email," the trigger word "email" matches.
INSTRUCTIONS (the body)
Step-by-step rules that get injected into the LLM's prompt. Write them like you're training a new employee — clear, specific, with examples. The better your instructions, the better the output.
EXAMPLES (optional)
Concrete input/output examples help the LLM learn the pattern. Especially useful for formatting-sensitive tasks (emails, reports, code style).

Writing your own skill in 3 steps

1
Notice a pattern
You keep giving the same instructions over and over — "write it in this format," "check these things first," "follow these rules."
2
Write a SKILL.md
Put those instructions in a SKILL.md file with a YAML header. Add triggers, examples, and edge cases. Use the Skill Creator skill to help you write it.
3
Test and iterate
Load the skill, try it with real tasks, and refine. Good skills go through 3–5 rounds of revision before they're solid. Publish to ClawHub when it's ready.
💡 THE META TRICK
Use the Skill Creator skill to write skills. Tell it "I want a skill that helps me write weekly reports" and it will generate the SKILL.md for you, including frontmatter, triggers, and examples. Then iterate on the output. It's skills all the way down.
LOCKING IT DOWN
06

Security Hardening

Your agent has access to your computer — treat it accordingly

OpenClaw runs on your machine with significant permissions. A misconfigured instance is an open door. These aren't optional suggestions — they're the minimum for safe operation.

🔒
Never run as root
Create a dedicated non-root user for OpenClaw with minimal permissions. If the agent gets compromised, the blast radius is limited to that user's access.
🔐
Lock down your secrets
Use environment variables for API keys and tokens — never put them in config files. Set your state directory to chmod 700 so only the OpenClaw user can read secrets and sessions.
🌐
Never expose to the public internet
Don't bind to 0.0.0.0 and don't set up a public reverse proxy. OpenClaw is not hardened for public exposure. If you need remote access, use an SSH tunnel or Tailscale.
🧩
Vet every skill
Only load skills you explicitly trust. Read the source code of any third-party skill before enabling it. Skills declare what permissions they need — review and approve each one explicitly.
🧠
Use instruction-hardened models
Claude Sonnet has the best prompt-injection resistance of any current model. For an agent with tool access, this matters — a model that can be tricked by a malicious email is a model that can be weaponized.
📋
Add safety boundaries to SOUL.md
Your soul file should include explicit boundaries: "Never execute shell commands without approval. Never send money. Never delete files permanently. Never share my credentials." The LLM follows what the soul says — so tell it.
⚠️ THE UNCOMFORTABLE TRUTH
No amount of hardening makes OpenClaw 100% safe. You are giving an AI agent access to your computer, your email, your files, and your accounts. The agent is only as trustworthy as the LLM driving it — and LLMs can be tricked. Use OpenClaw with the same caution you'd use when giving a new employee access to your systems: start with limited permissions and expand slowly as trust builds.

07

Daily Habits for Power Users

The difference between "I tried OpenClaw" and "OpenClaw changed how I work"
🌅
MORNING
Start with a Check-In
Text your agent "morning" or "what's up today?" first thing. A good HEARTBEAT.md means it already checked your email and calendar. You get a brief of what needs attention — before you open a single app.
🔄
WEEKLY
Review Your Memory Files
Skim memory/ once a week. Delete junk entries that add noise. Pin important context that keeps getting forgotten. Your agent's memory is only as clean as you keep it.
🎯
ONGOING
Iterate on SOUL.md
When the agent does something you don't like, don't just correct it once — update SOUL.md so it never does it again. Each correction is a permanent improvement. Some users tweak weekly; others monthly.
📊
ONGOING
Watch Your Token Usage
Monitor API costs. If your agent burns through tokens on simple tasks, you might need to tune model routing (use cheaper models for easy stuff) or trim your SOUL.md and memory files to reduce prompt size.
🎯 THE ONE HABIT THAT MATTERS MOST
Every time the agent does something wrong, ask yourself: "Is this a soul problem, a skill problem, or a model problem?" Soul problem → update SOUL.md. Skill problem → find a better skill or configure the current one. Model problem → switch to a stronger model for that task. This mental framework turns frustration into systematic improvement.
SCALING UP
08

Multi-Agent Orchestra

Why one agent is good, but multiple agents working together is transformational

A single OpenClaw agent can do a lot. But real power comes when you run multiple specialized agents that coordinate — like a team where each member has a different job. This is the "multi-agent orchestra" pattern.

Why multiple agents?

🎯
Specialization
A "research agent" with a long SOUL.md focused on thoroughness works better than a generalist asked to research. Each agent has its own soul, memory, and model tuned for one job.
🧱
Isolation
If your coding agent goes haywire, it doesn't affect your email agent. Separate agents mean separate failure domains, separate memory, and separate permissions.
💰
Cost Efficiency
Your email summarizer runs on cheap Gemini Flash. Your research agent runs on Opus. You only pay for expensive models where they're actually needed.

How it works in practice

Conductor Agent
→
Research Agent
Coding Agent
Email Agent
Calendar Agent
A conductor agent (your main OpenClaw instance) receives your message and decides which specialist to delegate to. You say "research competitors and draft an email to the team about it" — the conductor sends the first part to the research agent (running Opus for depth), waits for the result, then passes it to the email agent (running Sonnet for tone) to compose the message. Each agent has its own SOUL.md, its own skills, and its own model.
💡 START SIMPLE
Don't build an orchestra on day one. Start with a single well-configured agent. Once it's working well, fork it — duplicate the workspace, specialize the SOUL.md, assign a different model, and connect them. Most users find that 2–3 specialized agents (e.g., daily assistant + research + coding) hit the sweet spot before complexity outweighs the benefit.
